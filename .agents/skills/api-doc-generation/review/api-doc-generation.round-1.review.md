# 问题清单

## 问题 1

- 级别：中
- 位置：`SKILL.md` 的参数和返回字段规则
- 问题描述：原文要求“通过查看代码确认其约束”，但没有把“证据等级”和“证据定位”定义为输出必填项。
- 为什么危险：执行者可能只在脑中完成核对，不在文档中留下可复核证据，后续复审和争议定位会很难。
- 可能触发方式：共享 DTO 很大、service 调用链较长、约束分散在多个类里时，执行者容易省略追踪结果。
- 修复建议：把 `confirmed`、`inferred`、`uncertain` 和“证据定位”列写成强制输出项。
- 替代写法：要求每个关键字段都记录入口方法、字段使用点、校验点或返回组装点。

## 问题 2

- 级别：中
- 位置：`SKILL.md` 的整体结构
- 问题描述：原文没有单独的 `Quick Reference`，也没有显式声明当前仍缺少 RED -> GREEN -> REFACTOR 证据链。
- 为什么危险：后续阅读者不容易快速抓住执行约束，也容易把“文档写得更完整”误解成“已经通过 writing 验证”。
- 可能触发方式：skill 被复用或被纳入更大治理流程时，读者跳读正文而遗漏边界条件。
- 修复建议：补充扫描型 `Quick Reference`，同时增加 `Verification Status` 说明。
- 替代写法：用表格列出收敛范围、字段提取、依赖分析、不确定项等最低要求。

## 问题 3

- 级别：中
- 位置：`references/output-template.md`
- 问题描述：接口关系模板主要是叙述句，缺少上游接口、下游接口、关键字段、前置条件、是否转换等结构化列。
- 为什么危险：前端依赖关系容易写得抽象，无法直接拿来检查字段传递是否闭环。
- 可能触发方式：列表、详情、编辑、导出等多个接口串联时，字段传递点增多。
- 修复建议：把接口关系部分改成表格，并补充“关键字段”“前置条件”“是否需要转换”列。
- 替代写法：新增“接口依赖总览”总表，统一按前端动作汇总。
