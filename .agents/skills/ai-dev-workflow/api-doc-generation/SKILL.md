# 接口文档生成

## Overview

用于生成面向前端、测试、后端评审的人类可读接口文档，默认输出 Markdown 或 Wiki 风格文档。

核心原则：

- 接口文档是团队协作契约，不是 DTO、BO、VO 字段清单。
- 单接口明细按 Diataxis Reference 思路写事实：准确、结构固定、少解释。
- 页面或业务调用流程按 Diataxis How-to 思路写步骤：围绕用户要完成的任务，说明接口顺序、字段传递、成功后动作和失败处理。
- 字段、约束、返回、错误必须来自真实 controller、route、service、DTO、VO、mapper、converter、异常处理逻辑；不要机械抄复用对象。
- OpenAPI、JSON Schema、HTTP 语义、Problem Details 只作为质量检查参考；除非用户明确要求，不把 YAML 或 JSON 作为默认交付物。

## When To Use

- 用户指定一个或多个模块，希望梳理可读接口文档。
- 用户需要给前端做联调、接口调试、Mock、测试数据或页面串联说明。
- 用户需要确认请求参数、返回参数、枚举值、校验规则、默认值、分页规则、时间格式和字段约束。
- 用户需要说明页面或业务调用流程，例如列表 -> 详情 -> 编辑 -> 保存 -> 刷新列表。
- 用户需要整理错误响应、业务错误码、HTTP 状态码、权限不足、空数据、重复提交或幂等性行为。
- 用户需要接口版本记录、废弃说明、兼容性影响或前后端待确认项。
- 代码里存在复用 BO、统一 DTO、继承字段、分页对象，需要逐个接口裁掉多余字段。

## When Not To Use

- 用户只要泛化接口设计建议，不需要基于现有代码落文档。
- 后端代码不存在，只有口头需求。
- 用户只要 Swagger、Knife4j、Apifox 或 OpenAPI 导出结果，不要求代码核验和字段裁剪。
- 用户明确要求生成机器可读契约文件且不需要人类可读说明；这时按用户指定规范输出，不套用本模板。
- 目标不是 HTTP/REST 接口，例如纯 MQ、Kafka、WebSocket、gRPC、GraphQL；除非用户只要求按当前代码写人类可读说明。

## Quick Reference

| 任务 | 最低要求 |
| --- | --- |
| 收敛范围 | 先确定模块、目录、包路径、controller、route 或 handler 聚合点 |
| 默认交付 | Markdown/Wiki 风格可读文档，OpenAPI 只作可选附录 |
| 文档结构 | 模块概览、通用约定、页面调用流程、接口清单、接口详情、错误码、版本记录、待确认项 |
| 单接口明细 | 按 Reference 写事实表：方法、路径、鉴权、参数、响应、错误、边界、证据 |
| 调用流程 | 按 How-to 写目标导向步骤：前端动作、调用接口、依赖字段、成功后动作、失败处理 |
| 请求字段 | 只保留当前接口真实读取、校验、传递、组装查询条件的字段 |
| 返回字段 | 追踪 assembler、converter、mapper、脱敏、权限裁剪，确认真实返回结构 |
| 错误行为 | 写清 HTTP 状态码、业务错误码、触发场景、前端处理建议和证据 |
| 字段证据 | 每个关键字段标注 `confirmed`、`inferred` 或 `uncertain`，并给出证据定位 |
| 不确定项 | 证据不足时标记 `uncertain`，不要假装确认 |

## Required Inputs

最少获取这些输入：

- 目标模块、子系统、包路径、目录或关键词。
- 需要梳理的接口范围。
- 文档受众：前端联调、测试用例、后端评审，或三者都要。
- 如果范围不清楚，先按模块入口类、路由、controller、handler、consumer 收敛。

如果用户没有给出完整范围，先补齐最小问题：

- 模块名或代码路径是什么。
- 只看查询类接口，还是增删改查都要。
- 是否需要页面调用流程、错误码、版本记录和 Mock/测试数据。

## Documentation Shape

### 1. Reference：单接口事实说明

接口详情必须像参考手册一样稳定、可查、可复核。写清事实，不写长篇教学。

每个接口至少包含：

- 接口名称、用途、状态、负责人或归属模块、版本。
- Method、URL、环境地址或域名前缀、Content-Type、鉴权、是否登录。
- Header、Path、Query、Body 参数。
- 成功响应的 HTTP 状态码、响应示例和字段表。
- 错误响应的 HTTP 状态码、业务错误码、触发场景和前端处理建议。
- 空数据、权限不足、参数非法、重复提交、幂等性、分页、时间格式等边界规则。
- 代码入口、关键字段证据、待确认项。

### 2. How-to：页面或业务调用流程

调用流程必须从前端或业务目标出发，而不是从后端方法调用链出发。

每个流程至少说明：

- 用户或页面要完成的目标。
- 前置条件，例如已登录、已有记录、具备权限。
- 调用步骤，例如进入页面、查询列表、点击详情、提交保存、刷新列表。
- 每一步调用哪个接口、依赖上一步的哪些字段、字段名是否需要转换。
- 成功后前端应该做什么，失败时如何判断和处理。
- 哪些步骤可以并行，哪些步骤必须串行。

### 3. 规范参考不是默认输出

- REST/HTTP API 可参考 OpenAPI 的信息项，但默认仍输出可读 Markdown。
- JSON 字段约束可参考 JSON Schema 的类型、必填、枚举、格式、长度、范围表达。
- HTTP 方法和状态码语义按 HTTP 规范检查，不用业务错误码替代 HTTP 状态码。
- 错误响应可参考 Problem Details 的思想：错误要有类型、状态、可读信息、定位信息。
- 用户明确要求 OpenAPI、Swagger、Knife4j、Apifox、Postman 时，再生成或补充对应格式。

## Core Rules

### 1. 先找真实入口，再写接口清单

优先从这些位置定位接口入口：

- controller
- route 或 router
- handler
- RPC facade
- consumer
- message handler

每个接口至少确认：

- 接口类型，例如 `GET`、`POST`、`PUT`、`DELETE`。
- 接口路径。
- 入口文件和方法。
- 请求对象。
- 返回对象。
- 异常处理或错误码来源。

### 2. 先写模块约定，再写单接口

不要在每个接口里重复大量通用内容。先整理模块或系统级约定：

- 环境地址、域名前缀、base path。
- 鉴权方式，例如 Token、Cookie、OAuth2、签名、登录态。
- 通用 Header，例如 `Authorization`、`Content-Type`、`Accept-Language`、租户 ID、trace ID。
- 统一响应包装，例如 `code`、`message`、`data`、`traceId`。
- 分页规则，例如 `pageNum/pageSize`、`page/pageSize`、`cursor/limit`。
- 时间格式和时区，例如 ISO 8601、时间戳、`yyyy-MM-dd HH:mm:ss`。
- 枚举表达、布尔值表达、文件上传、导出下载、幂等键、重复提交规则。

如果代码证据不足，写 `未发现统一约定` 或标记 `uncertain`。

### 3. 只记录当前接口真实使用的请求字段

对每个接口，按下面顺序判断字段是否应该出现在文档里：

1. 找到入口方法的 path 参数、query 参数、header、body 对象。
2. 查看参数绑定、注解、校验器、schema、DTO、BO、VO 定义。
3. 继续追踪 service、use case、repository、query builder、converter。
4. 只保留当前接口真正读取、校验、传递、组装查询条件或更新内容时使用到的字段。

以下字段不要写进文档：

- 只是因为复用 BO 才存在，但当前接口没有读取的字段。
- 只在别的接口里使用的字段。
- 只在对象定义里声明，但没有参与校验、判断、查询、更新或返回组装的字段。

### 4. 约束必须来自代码，不要猜

优先从这些位置提取约束：

- 枚举类。
- validation annotation。
- schema 定义。
- 手写 `if` 判断。
- 正则校验。
- 长度限制。
- 数值范围限制。
- 非空判断。
- 默认值处理。
- 白名单转换。

文档里要明确写出：

- 字段类型。
- 是否必填。
- 允许值。
- 校验规则。
- 默认值。
- 示例值。

约束表达要求：

- 枚举类：列出所有允许值和含义。
- 长度限制：写明最小和最大长度。
- 数值范围：写明闭区间或开区间。
- 正则：用自然语言描述格式，并保留正则关键片段。
- 条件必填：写明触发条件，例如“当 `type=custom` 时必填”。

### 5. 返回字段同样只记录真实返回内容

不要只看 VO 或 Response 类定义。

必须继续确认：

- 返回对象是直接返回，还是经过 assembler、converter、mapper 二次组装。
- 是否有字段裁剪。
- 是否有脱敏。
- 是否根据权限、租户、状态决定是否返回某字段。
- 是否只在列表返回摘要字段，详情返回完整字段。
- 是否存在统一响应包装、分页包装或文件流响应。

返回字段说明至少包含：

- 字段名。
- 类型。
- 必填性或是否可能为空。
- 含义。
- 示例值。
- 取值约束或枚举值。
- 证据等级和证据定位。

### 6. 错误响应和边界行为必须单独写

每个接口都要检查是否存在这些行为：

- 参数非法。
- 未登录或登录失效。
- 无权限。
- 数据不存在。
- 数据状态不允许操作。
- 空列表或空对象。
- 重复提交。
- 幂等重试。
- 上传、导出、下载失败。
- 服务端异常。

文档中至少写清：

- HTTP 状态码。
- 业务错误码。
- 错误信息示例。
- 触发场景。
- 前端处理建议。
- 证据定位。

如果项目统一使用 `200` 包裹业务错误，也要如实记录，并标明业务错误码判断方式。

### 7. 每个关键字段都要留下证据定位

对请求字段、返回字段、约束字段、错误码、流程依赖字段，优先补下面两类证据：

- 证据等级：`confirmed`、`inferred`、`uncertain`。
- 证据定位：类名、方法名、注解、校验器、查询构造点、返回组装点、异常处理点。

推荐粒度：

- 控制器或路由入口。
- DTO 或 schema 定义。
- service/use case 中的字段使用点。
- repository/query builder 中的查询条件拼装点。
- converter/assembler 中的返回字段组装点。
- exception handler 或错误码枚举。

### 8. 页面调用流程必须贴近前端操作

除了单个接口说明，还要补“页面/调用流程”。

重点分析这些关系：

- 列表 -> 详情。
- 列表 -> 编辑。
- 新建/编辑 -> 详情。
- 新建/编辑/删除 -> 列表刷新。
- 上传 -> 提交。
- 主数据查询 -> 下拉选项或枚举查询。
- 条件查询 -> 导出。
- 父记录 -> 子记录列表。
- 草稿保存 -> 提交审核 -> 审核详情。

每条关系都要说明：

- 前端动作是什么。
- 上游接口产出什么字段。
- 下游接口依赖哪些字段。
- 字段名是否一致，是否需要转换。
- 是否存在状态或权限前置条件。
- 成功后是否刷新、跳转、关闭弹窗或保留筛选条件。
- 失败时前端依据什么字段展示提示。

## Working Procedure

### Step 1. 收敛模块范围

- 定位模块目录、包路径、controller 或 route 聚合点。
- 列出模块下的候选接口。
- 排除与当前模块无关的通用接口。
- 查找项目内已有接口文档、错误码文档、统一响应说明、前端页面名。

### Step 2. 整理模块通用约定

- 记录环境地址、base path、鉴权、通用 Header、响应包装、分页、时间格式、trace、租户。
- 如果没有代码或文档证据，标记 `uncertain`。

### Step 3. 建立页面或业务调用流程

- 按前端页面或业务任务排序，而不是按 controller 文件顺序。
- 为每个流程写目标、前置条件、步骤、接口、字段依赖、成功动作、失败处理。
- 不知道前端页面时，按后端接口自然流程推断，并标记 `inferred`。

### Step 4. 逐个接口建 Reference 卡片

每个接口先记录：

- 接口名称。
- 接口类型。
- 接口路径。
- 用途说明。
- 状态、版本、负责人或归属模块。
- 入口文件和方法。
- 请求对象。
- 返回对象。
- 错误码或异常处理来源。

### Step 5. 提取请求参数

按 `header`、`path`、`query`、`body` 分组。

查询类接口重点核对：

- 分页参数是否真的生效。
- 排序字段是否有限制。
- 筛选字段是否真正参与查询条件。
- 状态、类型、日期范围是否有枚举或格式要求。

写入接口文档前，确认字段是否在当前接口真实使用。

### Step 6. 提取返回参数

至少区分：

- 列表接口返回的行对象。
- 详情接口返回的完整对象。
- 操作类接口返回的结果对象。
- 分页包装结构，例如 `total`、`pageNo`、`pageSize`、`records`。
- 文件流、下载地址、异步任务 ID 等特殊返回。

如果返回结构存在通用包装，只写当前接口实际返回层级，不要把所有全局包装字段机械展开成业务字段。

### Step 7. 补错误、边界和版本记录

- 查找全局异常处理、业务异常、错误码枚举、权限注解、状态校验。
- 为接口补齐常见错误和边界行为。
- 如果接口新增、废弃或变更，写入版本记录和兼容性影响。

### Step 8. 输出最终文档

输出格式使用 [references/output-template.md](references/output-template.md)。

## Output Contract

文档必须按模块输出，不能只给零散字段。

模块文档至少包含：

- 模块概览。
- 通用约定。
- 页面或业务调用流程。
- 接口清单。
- 单接口详情。
- 错误码说明。
- 版本变更记录。
- 风险与待确认项。

每个接口至少包含：

- 接口名称、用途、状态、版本、负责人或归属模块。
- 请求信息：Method、URL、Content-Type、Auth、是否登录、环境或 base path。
- 请求参数：Header、Path、Query、Body。
- 成功响应：HTTP 状态码、响应示例、字段说明。
- 错误响应：HTTP 状态码、业务错误码、触发场景、前端处理建议。
- 边界规则：空数据、无权限、重复提交、幂等性、分页、时间格式等。
- 调用示例。
- 与其他接口或页面流程的关系。
- 证据和待确认项。

字段说明必须包含：

- 字段名。
- 位置。
- 类型。
- 必填性或是否可能为空。
- 示例值。
- 约束。
- 约束来源。
- 证据等级。
- 证据定位。
- 说明。

证据等级使用：

- `confirmed`: 直接在代码中看到定义、校验、使用或返回组装。
- `inferred`: 从调用链可高置信推断，但没有单点明文定义。
- `uncertain`: 代码证据不足，需要人工确认。

如果某个接口没有 header、path、query、body、错误响应、边界规则或依赖关系，必须明确写 `无` 或 `未发现`，不要省略。

## Quality Bar

合格文档至少满足：

- 前端不问后端，也能知道地址、鉴权、参数、响应、错误判断和调用顺序。
- 测试不找人确认，也能写成功、失败、空数据、无权限、参数非法等用例。
- 后端不解释源码，也能复核接口行为和字段来源。
- 接口变更后，影响页面、字段和下游接口可追踪。
- 示例值能直接用于 Postman、Apifox 或浏览器调试。

## Common Mistakes

- 用户需要可读文档时，默认输出 OpenAPI YAML/JSON。
- 直接把复用 BO、DTO、VO 的全部字段抄进接口文档。
- 只看参数类定义，不追踪字段是否真的被使用。
- 只看返回类定义，不确认是否经过转换、脱敏或裁剪。
- 只写单接口参数，不写页面调用流程。
- 调用流程按 controller 顺序写，而不是按前端任务顺序写。
- 漏掉 header、path 参数、鉴权、环境地址或 Content-Type。
- 枚举字段没有展开允许值。
- 有校验注解却没写校验规则。
- 错误响应只写“失败”，没有 HTTP 状态码、业务错误码和触发场景。
- 把业务错误码当成 HTTP 状态码。
- 关键字段没有留下证据定位。
- 把全局分页对象的所有字段都写成当前接口业务字段。
- 把推断内容写成确认事实，没有标记 `inferred` 或 `uncertain`。

## Verification Status

当前版本是基于实际使用反馈和静态规范整理的 round-2 优化版，已明确可读文档优先、Diataxis Reference/How-to 组织、错误码和版本记录要求。

当前版本尚未完成 `writing-skills` 所要求的完整 RED -> GREEN -> REFACTOR 多代理压力验证，因此只能宣称 `static-compliant`，不能宣称完整 writing 合规版本。

## Quick Checklist

在交付前逐项确认：

- 是否已经按模块收敛接口范围。
- 是否整理了环境、鉴权、Header、响应包装、分页、时间格式等通用约定。
- 是否按页面或业务任务写了调用流程。
- 是否每个接口都确认了类型、路径、入口方法、请求对象、返回对象。
- 是否每个请求字段都确认“当前接口真的使用”。
- 是否每个关键返回字段都有类型、示例、必填性或可空性。
