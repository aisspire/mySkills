# 接口文档生成

## Overview

生成让前端“几乎无需再向后端确认”的接口交付文档。

核心原则：

- 文档描述当前代码真实行为，不描述理想中的接口。
- 请求、响应、枚举、校验和业务规则必须由代码确认。
- 前端文档优先解释页面目标、调用顺序、字段传递和失败处理。
- 通用约定只写一次，单接口只写差异。
- 更新已有文档时，版本摘要和兼容性差异是首要内容。
- 不在正文堆叠代码证据位置、负责人和维护状态。

## When to Use

- 用户明确要求产出或更新 API 文档、接口文档或前端联调文档。
- 用户要求核验接口请求字段、返回字段、枚举、约束或版本差异。
- 用户要求把后端接口整理成可直接交付前端、测试或外部团队的文档。

## When NOT to Use

- 普通代码修改后的例行文档同步。
- 只需要维护 `docs/api/` 的轻量接口地图。
- 用户只想理解某段后端逻辑，没有要求接口交付物。
- 只有口头需求、没有现有接口实现，并且用户要的是接口设计。
- 用户只要求导出 OpenAPI、Swagger、Knife4j、Apifox 或 Postman 文件。

## Output Position

优先沿用项目现有详细接口文档位置和命名。

如果项目没有约定：

- 模块级交付：`docs/<domain>/<module>/接口文档.md`
- 跨模块交付：`docs/delivery/api/<topic>.md`

不要用完整前端文档替换 `docs/api/<module>.md` 的轻量接口地图；在接口地图中添加链接即可。

## Source Policy

事实优先级：

1. 当前 Controller、路由或公开接口入口。
2. 当前请求/响应类型、校验、枚举和序列化配置。
3. 当前 Service、Converter、Mapper 和异常处理中的实际行为。
4. 现有接口文档，仅用于版本对比和历史变化。

字段规则：

- 每个写入请求表和响应表的字段都必须由代码确认。
- 不使用逐字段 `confirmed/inferred/uncertain` 标记。
- 不添加类名、方法名、文件路径或行号证据列。
- 外部系统行为、环境地址、测试账号或业务口径无法确认时，放入文档末尾“待确认项”。
- 代码疑似错误时，如实记录当前行为，并在“后端问题与联调风险”中说明期望、实际和影响。

## Progressive Code Inspection

按需深入，避免一次读取完整模块：

### Level 1：接口地图

读取 Controller、route 或 handler，确认：

- Method 和 Path。
- 请求位置和请求对象。
- 返回类型。
- 权限、重复提交、导入导出等入口注解。

### Level 2：数据契约

读取本次接口涉及的 BO、DTO、VO、分页对象、枚举和校验，确认：

- 实际请求字段。
- 完整业务响应结构。
- 类型、必填、可空、默认值、格式、枚举和范围。

### Level 3：业务语义

只有字段用途、状态变化、默认行为、错误或返回组装仍不清楚时，再读取相关 Service、Converter、Mapper 和异常逻辑。

### Level 4：公共约定

鉴权、响应包装、分页、时间格式和公共异常只核对一次，在模块通用契约中统一说明。

确认当前文档所需事实后停止扩展调用链。

## Documentation Shape

### 1. 版本变更摘要

更新已有文档时放在靠前位置，按表格说明：

- 新增、修改、删除。
- 兼容或破坏性变化。
- 前端需要执行的动作。

初版文档可以省略差异摘要，只保留版本记录。

### 2. 前端快速开始

集中说明：

- Base path、鉴权和 Content-Type。
- 统一响应包装。
- 分页、时间、枚举和空值约定。
- 页面入口和推荐调用顺序。

无法从代码确认的环境域名、账号和网关前缀不要猜。

### 3. 页面或业务流程

用自然语言说明用户目标、前置条件、接口顺序、字段传递、成功动作和失败处理。

按复杂度选图：

| 场景 | 推荐表达 |
| --- | --- |
| 简单两步调用 | 自然语言或编号步骤 |
| 三步以上、有分支或异步 | Mermaid `flowchart` |
| 多角色或跨服务调用 | Mermaid `sequenceDiagram` |
| 明确状态迁移 | Mermaid `stateDiagram-v2` |

图必须配自然语言，不能只交付图。

### 4. 接口清单

按页面任务或业务流程排序，不按 Controller 文件顺序排序。

至少列出：

- 接口名称。
- Method 和 Path。
- 页面用途。
- 请求对象或关键输入。
- 返回数据。

### 5. 单接口详情

每个接口保留前端真正需要的内容：

- 用途和调用时机。
- Method、Path 和特殊请求格式。
- Path、Query、Body、特殊 Header 参数。
- 请求示例。
- 成功响应示例。
- 完整业务返回字段。
- 当前代码中的业务规则和边界。
- 当前代码中真实存在且前端需要处理的错误。

没有某类参数时可以省略该小节，不强制写“无”。

### 6. 枚举和公共字典

多个接口重复使用的枚举集中说明一次。条件必填和状态限制要写清触发条件。

### 7. 后端问题与联调风险

记录：

- 代码行为与显然业务期望不一致。
- 请求对象接收字段但实际忽略。
- 返回字段命名或类型不一致。
- 状态、空值、权限或错误行为会增加前端复杂度。
- 文档无法确认的外部依赖。

不要让前端用大量兼容逻辑掩盖后端缺陷；优先把问题暴露为待修复项。

### 8. 版本记录

保留每次交付版本、日期、变化范围、兼容性和前端动作。

## Request Contract Rules

- 只记录当前接口实际接受并使用的字段。
- 复用 BO 中未参与当前接口逻辑的字段不进入文档。
- 字段表包含：字段、位置、类型、必填、默认值、约束、示例、说明。
- 枚举列出允许值和含义。
- 条件必填写出触发条件。
- 后端忽略前端提交值时明确标为“只读/由后端生成”。
- 多值 Query、数组、文件上传和下载给出可直接复制的调用示例。

## Response Contract Rules

- 返回参数必须完整到足以渲染页面、回填表单和串联下一接口。
- 先确认统一包装和分页层级，再写业务数据。
- 区分列表、详情、操作结果、文件流和异步任务。
- 字段表包含：字段、类型、可空、示例、说明或枚举。
- 返回对象经过转换、聚合、脱敏或权限裁剪时，记录最终对外结构。
- 不把只存在于 VO 定义、但实际不会序列化或不会赋值的字段当成稳定契约。

## Errors and Boundaries

只记录真实且前端可行动的错误：

- 业务错误码或稳定错误消息。
- 触发条件。
- 前端处理动作。

公共 401、403、500 等行为若由全局约定统一处理，只在通用契约写一次。

检查边界是生成文档时的后端审查步骤。若代码没有形成稳定行为：

- 不虚构接口契约。
- 在“后端问题与联调风险”中记录。
- 明确是否会阻塞前端联调。

## Version Update Procedure

更新已有文档时：

1. 读取当前文档版本和最近变更摘要。
2. 以当前代码重新确认受影响接口。
3. 比较请求字段、返回字段、枚举、规则、错误和流程。
4. 将变化分类为新增、兼容修改、破坏性修改、删除或仅实现变化。
5. 写清前端是否需要改参数、响应解析、页面流程、校验或错误处理。
6. 更新正文中的当前契约。
7. 追加版本记录，不把历史差异混入当前字段表。

## Output Resource

使用 [references/output-template.md](references/output-template.md)。

按项目实际复杂度删除不适用章节，不要为了填模板制造空内容。

## Quality Bar

交付前确认：

- 前端能找到所有页面所需接口和调用顺序。
- 请求字段、返回字段和枚举足以完成联调。
- 响应示例与字段表一致。
- 当前版本变化和前端动作清楚。
- 业务规则、状态和关键边界使用自然语言解释。
- 复杂流程在需要时有图，简单流程没有多余图。
- 正文没有代码证据列、负责人和大量重复公共信息。
- 代码缺陷和外部未确认项没有被伪装成稳定契约。

## Common Mistakes

- 普通开发任务自动生成或重写完整接口文档。
- 只列 Method 和 Path，不写完整返回参数。
- 机械复制 BO、DTO、VO 全部字段。
- 每个字段附代码位置和证据等级。
- 每个接口重复鉴权、分页和公共错误。
- 用假设补齐代码未实现的错误和边界。
- 把后端缺陷转化为前端兼容要求。
- 只画流程图，不用自然语言说明业务目的和约束。
- 更新正文却不写版本差异和前端动作。
