# 接口文档生成

## Overview

用于给前端联调、接口调试、页面串联说明生成可直接使用的接口文档。

核心原则：只写当前接口在代码里真实使用到的字段、约束和依赖关系，不要把复用 DTO、BO、VO 的全部字段机械抄入文档。

## When To Use

- 用户指定一个或多个模块，希望梳理接口文档
- 用户需要给前端做联调、调试、Mock 或页面交互说明
- 用户需要确认请求参数、返回参数、枚举值、校验规则、默认值和字段约束
- 用户明确要求说明接口之间的依赖关系，例如列表到详情、保存后刷新列表、上传后提交
- 代码里存在复用 BO、统一 DTO、继承字段、分页对象，需要逐个接口裁掉多余字段

## When Not To Use

- 用户只要泛化接口设计建议，不需要基于现有代码落文档
- 后端代码不存在，只有口头需求
- 用户只要 OpenAPI 或 Swagger 导出结果，不要求代码核验和字段裁剪

## Quick Reference

| 任务 | 最低要求 |
| --- | --- |
| 收敛范围 | 先确定模块、目录、包路径、controller 或 route 聚合点 |
| 识别接口 | 每个接口至少确认类型、路径、入口方法、请求对象、返回对象 |
| 请求字段 | 只保留当前接口真实读取、校验、传递的字段 |
| 返回字段 | 继续追踪 assembler、converter、mapper，确认真实返回结构 |
| 字段约束 | 枚举、长度、范围、正则、非空、默认值都要落到文档 |
| 字段证据 | 每个关键字段都要标注证据等级和证据定位 |
| 接口关系 | 用前端动作描述上游接口、下游接口、关键字段和前置条件 |
| 不确定项 | 证据不足时标记 `uncertain`，不要假装确认 |

## Required Inputs

最少获取这些输入：

- 目标模块、子系统、包路径、目录或关键词
- 需要梳理的接口范围
- 如果范围不清楚，先按模块入口类、路由、controller、handler、consumer 收敛

如果用户没有给出完整范围，先补齐最小问题：

- 模块名或代码路径是什么
- 只看查询类接口，还是增删改查都要
- 结果是给前端调试，还是也要给测试或后端评审

## Core Rules

### 1. 先找真实入口，再写接口清单

优先从这些位置定位接口入口：

- controller
- route 或 router
- handler
- RPC facade
- consumer
- message handler

每个接口至少确认：

- 接口类型，例如 `GET`、`POST`、`PUT`、`DELETE`
- 接口路径
- 入口文件和方法
- 请求对象
- 返回对象

### 2. 只记录当前接口真实使用的请求字段

对每个接口，按下面顺序判断字段是否应该出现在文档里：

1. 找到入口方法的 path 参数、query 参数、header、body 对象
2. 查看参数绑定、注解、校验器、schema、DTO、BO、VO 定义
3. 继续往下追踪 service、use case、repository、query builder、converter
4. 只保留当前接口真正读取、校验、传递、组装查询条件时使用到的字段

以下字段不要写进文档：

- 只是因为复用 BO 才存在，但当前接口没有读取的字段
- 只在别的接口里使用的字段
- 只在对象定义里声明，但没有参与校验、判断、查询、更新或返回组装的字段

### 3. 约束必须来自代码，不要猜

优先从这些位置提取约束：

- 枚举类
- validation annotation
- schema 定义
- 手写 `if` 判断
- 正则校验
- 长度限制
- 数值范围限制
- 非空判断
- 默认值处理
- 白名单转换

文档里要明确写出：

- 字段类型
- 是否必填
- 允许值
- 校验规则
- 默认值
- 示例值

约束表达要求：

- 枚举类：列出所有允许值和含义
- 长度限制：写明最小和最大长度
- 数值范围：写明闭区间或开区间
- 正则：用自然语言描述格式，并保留正则关键片段
- 条件必填：写明触发条件，例如“当 `type=custom` 时必填”

### 4. 返回字段同样只记录真实返回内容

不要只看 VO 或 Response 类定义。

必须继续确认：

- 返回对象是直接返回，还是经过 assembler、converter、mapper 二次组装
- 是否有字段裁剪
- 是否有脱敏
- 是否根据权限或状态决定是否返回某字段
- 是否只在列表返回摘要字段，详情返回完整字段

返回字段说明至少包含：

- 字段名
- 类型
- 含义
- 示例值
- 取值约束或枚举值

### 5. 每个关键字段都要留下证据定位

对请求字段、返回字段、约束字段，优先补下面两类证据：

- 证据等级：`confirmed`、`inferred`、`uncertain`
- 证据定位：类名、方法名、注解、校验器、查询构造点、返回组装点

推荐粒度：

- 控制器或路由入口
- DTO 或 schema 定义
- service/use case 中的字段使用点
- repository/query builder 中的查询条件拼装点
- converter/assembler 中的返回字段组装点

### 6. 关系分析必须贴近前端操作流

除了单个接口说明，还要补“接口之间的依赖关系”。

重点分析这些关系：

- 列表 -> 详情
- 列表 -> 编辑
- 新建/编辑 -> 详情
- 新建/编辑/删除 -> 列表刷新
- 上传 -> 提交
- 主数据查询 -> 下拉选项或枚举查询
- 条件查询 -> 导出
- 父记录 -> 子记录列表
- 草稿保存 -> 提交审核 -> 审核详情

每条关系都要说明：

- 前端动作是什么
- 上游接口产出什么字段
- 下游接口依赖哪些字段
- 字段名是否一致，是否需要转换
- 是否存在状态前置条件

## Working Procedure

### Step 1. 收敛模块范围

- 先定位模块目录、包路径、controller 或 route 聚合点
- 列出模块下的候选接口
- 排除与当前模块无关的通用接口

### Step 2. 逐个接口建卡片

每个接口先记录：

- 接口名称
- 接口类型
- 接口路径
- 入口文件和方法
- 请求对象
- 返回对象

### Step 3. 提取请求参数

按 `path`、`query`、`body`、`header` 分组。

其中查询类接口重点看 `query` 参数，至少核对：

- 分页参数是否真的生效
- 排序字段是否有限制
- 筛选字段是否真正参与查询条件
- 状态、类型、日期范围是否有枚举或格式要求

### Step 4. 提取返回参数

至少区分：

- 列表接口返回的行对象
- 详情接口返回的完整对象
- 操作类接口返回的结果对象
- 分页包装结构，例如 `total`、`pageNo`、`pageSize`、`records`

如果返回结构存在通用包装，只写当前接口实际返回层级，不要把所有全局包装字段机械展开。

### Step 5. 画出依赖关系

从前端视角描述，而不是只从后端类调用描述。

推荐使用这种句式：

- 在列表页点击某一行“详情”时，前端从列表接口的 `id` 字段取值，传给详情接口的 path 参数 `id`
- 在提交编辑表单后，前端通常需要重新调用列表接口，并沿用当前筛选条件
- 在创建成功后，后端返回 `id`，前端可继续调用详情接口拉取完整数据

### Step 6. 输出最终文档

输出格式使用 [references/output-template.md](references/output-template.md)。

## Output Contract

文档必须按接口维度输出，不能只给零散字段。

每个接口至少包含：

- 接口名称
- 接口类型
- 接口路径
- 用途说明
- 请求参数
- 返回参数
- 约束说明
- 调用示例
- 与其他接口的关系

字段说明必须包含：

- 字段名
- 位置
- 类型
- 必填性
- 示例值
- 约束来源
- 证据等级
- 证据定位

约束来源使用这些标签：

- `confirmed`: 直接在代码中看到定义、校验或使用
- `inferred`: 从调用链可高置信推断，但没有单点明文定义
- `uncertain`: 代码证据不足，需要人工确认

如果某个接口没有 path 参数、query 参数、body、header 特殊要求或依赖关系，必须明确写 `无`，不要省略。

## Common Mistakes

- 直接把复用 BO 的全部字段抄进接口文档
- 只看参数类定义，不追踪字段是否真的被使用
- 只看返回类定义，不确认是否经过转换或裁剪
- 漏掉 path 参数，只写 query 参数
- 枚举字段没有展开允许值
- 有校验注解却没写校验规则
- 关键字段没有留下证据定位
- 把全局分页对象的所有字段都写成当前接口业务字段
- 只写后端调用链，不写前端交互依赖

## Quick Checklist

在交付前逐项确认：

- 是否已经按模块收敛接口范围
- 是否每个接口都确认了类型和路径
- 是否每个请求字段都确认“当前接口真的使用”
- 是否每个关键返回字段都有类型和示例
- 是否每个关键字段都补了证据等级和证据定位
- 是否枚举、长度、范围、正则、必填规则都已展开
- 是否明确列出接口之间的前端依赖关系
- 是否把不确定项标成 `uncertain`
