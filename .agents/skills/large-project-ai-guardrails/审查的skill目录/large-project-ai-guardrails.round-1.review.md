# 第 1 轮审查报告：large-project-ai-guardrails

## 审查范围

- 目标 skill：`large-project-ai-guardrails`
- 输入文件：`.agents/skills/large-project-ai-guardrails/SKILL.md`
- 审查重点：模板易用性、触发稳定性

## 基线情况

- 当前正文在本环境中不可读，导致这个 skill 模板实际上无法正常使用。
- `description` 方向基本正确，但触发词仍然过泛，面对 monorepo、遗留仓库、迁移任务、跨模块改动时不够稳定。
- 模板没有清楚区分“必须填写”和“可选说明”。
- 首屏没有明确告诉用户在使用前必须先补哪些信息，容易让模板停留在泛化状态。

## 风险评级

- 总体等级：`D`
- 严重问题数：`1`
- 高风险问题数：`3`

## 发现的问题

### 严重问题

- 正文不可读，直接阻断 skill 的实际使用。
  - 影响：人和模型都无法稳定理解并执行模板内容。
  - 原因：对于 skill 模板来说，这属于完全失效，因为工作流无法开始。

### 高风险问题

- 触发描述过宽。
  - 影响：在“不熟悉的大仓库”“monorepo”“大范围清理”“迁移”等具体场景下，skill 不够容易被稳定选中。
- 缺少快速填写入口。
  - 影响：用户不知道哪些字段必须先补齐，模板很容易保持空泛。
- 模板前半部分堆了太多泛化结构。
  - 影响：真正关键的项目边界信息没有足够早地暴露出来。

### 中风险问题

- 没有明确区分“可安全修改区域”和“禁止直接触碰区域”。
  - 影响：用户可能漏填最关键的边界信息。
- 验证部分没有落到可执行命令层面。
  - 影响：模板看起来完整，但实际执行时仍然不够具体。

## 必须修复项

- 恢复为可读、直白的 skill 正文。
- 将 `description` 改成只描述触发条件，不描述流程。
- 在首屏增加 `Quick Start`，明确必须填写的字段。
- 将边界信息显式拆分为：安全区域、禁区、审批点、验证命令。

## 建议优化方向

- 采用混合模板结构：
  - 前半部分用于提升触发率和快速上手
  - 后半部分用于填写项目专属 guardrails 和检查项
- 在用户明确发布前，候选版本与正式入口保持分离。

## 本轮目标结果

- 目标等级：`B`
- 未直接追求 `A` 的原因：该模板仍然依赖人工结合具体仓库信息填写，暂时无法自动保证完整性。
